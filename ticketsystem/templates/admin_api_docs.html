API-Dokumentation

Technische Referenz für die öffentliche REST-API (HalloPetra / externe Systeme).

Übersicht

Die REST-API ermöglicht externen Systemen (z.B. dem Chatbot HalloPetra), Tickets im Ticketsystem zu erstellen. Alle Endpunkte sind unter {{ api_base }}/api/v1/ erreichbar.

Authentifizierung erfolgt ausschließlich via Bearer-Token im HTTP-Header. Kein Cookie-basierter Zugriff, kein CSRF.

Authentifizierung

Jede Anfrage muss einen gültigen API-Schlüssel im Authorization-Header senden:

Authorization: Bearer tsk_<token>

Token-Format: tsk_ Präfix + 48 alphanumerische Zeichen. Ungültige oder widerrufene Token erhalten 401 Unauthorized.

POST /api/v1/tickets — Ticket erstellen

Erstellt ein neues Ticket. Erfordert Scope write:tickets.

Request-Header

Authorization: Bearer tsk_<token>
Content-Type: application/json
X-Request-ID: <uuid>              (optional, für Idempotenz)
X-Webhook-ID: <webhook-id>        (erforderlich wenn expected_webhook_id gesetzt)

Request-Body (JSON)

{
  "title": "Rückrufanfrage von Firma Mustermann",   // Pflicht, max. 200 Zeichen
  "description": "Kunde wünscht Rückruf wegen...",  // optional
  "priority": "mittel",                              // optional: niedrig|mittel|hoch
  "contact": {
    "name": "Max Mustermann",                        // optional
    "phone": "+49 89 123456",                        // optional
    "email": "max@example.com",                      // optional
    "channel": "telefon",                            // optional
    "callback_requested": true,                      // optional
    "callback_due": "2026-04-15T10:00:00"           // optional ISO-8601
  },
  "external_ref": "chat-session-42"                 // optional, max. 64 Zeichen
}

Response (201 Created)

{
  "success": true,
  "ticket_id": 1234,
  "ticket_url": "https://<domain>/ticket/1234",
  "assignment_method": "default_assignee",
  "request_id": "abc-123-..."
}

Idempotenz

Über den optionalen Header X-Request-ID: <uuid> kann ein Aufrufer sicherstellen, dass ein Request auch bei Netzwerk-Wiederholungen nicht doppelt verarbeitet wird.

  • Wird eine Request-ID ein zweites Mal gesendet, antwortet die API mit 409 Conflict.
  • Die Response enthält die ticket_id des bereits erstellten Tickets.
  • Request-IDs werden für 24 Stunden gespeichert und dann gelöscht.

Rate-Limiting

Jeder API-Schlüssel hat ein konfigurierbares Rate-Limit (Standard: 60 Anfragen/Minute). Bei Überschreitung wird 429 Too Many Requests mit dem Header Retry-After zurückgegeben. 

HTTP/1.1 429 Too Many Requests
Retry-After: 15

{"success": false, "error": "Rate limit exceeded"}

IP-Allowlist

Ist für einen Schlüssel eine IP-Allowlist konfiguriert, werden Anfragen von nicht eingetragenen IP-Adressen mit 403 Forbidden abgelehnt — unabhängig vom Token. Leere Allowlist = alle IPs erlaubt. CIDR-Notation wird unterstützt (z.B. 10.0.0.0/8).

Datenschutz (DSGVO)

Über die API übermittelte Kontaktdaten (Name, Telefon, E-Mail) werden als Ticket-Kontaktdaten gespeichert. Tickets, die via API erstellt werden, sind standardmäßig als vertraulich markiert — sichtbar nur für berechtigte Mitarbeiter.

Audit-Log-Einträge enthalten IP-Adressen und werden nach 90 Tagen automatisch gelöscht. Bitte stellen Sie sicher, dass Ihr Auftragsverarbeitungsvertrag (AVV) diese Verarbeitung abdeckt.

Anhang: DNS-Einrichtung für den Webadmin

Für die Einrichtung der API-Subdomain ist ein CNAME-Eintrag beim Webadmin notwendig. Die folgende Anleitung ist als Vorlage gedacht; bitte an den Webadmin weiterleiten:

Ziel

Eine neue Subdomain (z. B. ticket-api.eure-domain.de) soll auf den Cloudflare Tunnel des Ticketsystems zeigen.

Vorgehen

  1. Im DNS-Management der Domain einen CNAME-Eintrag anlegen:
    • Name: ticket-api
    • Ziel: <tunnel-id>.cfargotunnel.com (konkreten Wert nach Tunnel-Einrichtung erfragen)
    • TTL: 300 (5 Minuten) zunächst, nach Validierung auf 3600
  2. KEIN A-Record, KEIN MX-Record, KEINE Port-Weiterleitung. Cloudflare terminiert TLS und stellt das Zertifikat automatisch aus.
  3. Propagation prüfen: dig ticket-api.eure-domain.de CNAME

Sicherheit

Die Subdomain ist ausschließlich für die externe Webhook-Integration. Keine E-Mail-Einträge, keine weiteren Services auf dieser Subdomain. Bitte diese Subdomain nicht für andere Zwecke verwenden.
